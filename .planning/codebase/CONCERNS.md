# Concerns

## Tech Debt

### 1. Type safety gaps
- Several API route handlers use partial typing or implicit `any` in error handling
- Some `supabase.from()` query results are not fully typed against `Database` types from `lib/supabase/types.ts`

### 2. Duplicated badge color configuration
- Badge color logic (estado → color mapping) is duplicated across multiple page components instead of being centralized in a shared utility or component

### 3. Duplicated auth helpers
- Role-checking logic (`rol === 'admin'`) is repeated inline across API routes rather than extracted into a shared `requireAdmin(request)` helper

### 4. Duplicated duplicate-check logic
- Human-readable ID generation (counting existing records to pick next ID) is repeated across each POST route (`/api/clientes`, `/api/visas`, `/api/pagos`, `/api/seminarios`) rather than extracted to a shared utility

### 5. Duplicated terminal states constant
- FINALIZADO/CANCELADA cascade logic is duplicated in 3+ places

### 6. Inline styles vs Tailwind
- Some components mix inline `style={{}}` with Tailwind classes inconsistently

---

## Known Bugs

### 1. Wrong visa/pago state shown in client list
- Client list view may show incorrect visa or payment status due to aggregation logic

### 2. Bulk-delete is hard-delete instead of soft-delete
- Bulk deletion does not respect the soft-delete rule (should mark as INACTIVO, not DELETE)

### 3. Inline edit silently fails
- Certain inline edit interactions fail silently without user feedback on error

---

## Security

### 1. Missing timing-safe comparison
- `validateApiKey()` in `lib/auth-m2m.ts` likely uses string equality (`===`) instead of a timing-safe comparison, making it theoretically vulnerable to timing attacks

### 2. Incorrect HTTP status codes
- Some routes return `401 Unauthorized` where `403 Forbidden` is semantically correct (authenticated but insufficient role)

### 3. Webhook middleware bypass
- `middleware.ts` protects dashboard routes but webhook routes in `app/api/` rely solely on `validateApiKey()` — no secondary protection layer

### 4. `configuracion` table read open to all authenticated users
- The `configuracion` table (which stores `precio_visa`, `precio_seminario`) has no role restriction — any `colaborador` can read pricing config

---

## Performance

### 1. No pagination on clientes list
- `app/(dashboard)/clientes/` loads all records without pagination — will degrade as dataset grows

### 2. No pagination on tramites/visas list
- Same issue on visa and process list pages

### 3. No pagination on pagos list
- Same issue on payments list page

---

## Fragile Areas

### 1. Client detail page uses sequential queries
- `app/(dashboard)/clientes/[id]/` fires multiple Supabase queries sequentially in the server component instead of using `Promise.all()`, adding latency

### 2. FINALIZADO cascade duplicated in 3 places
- Logic that cascades a client state change to FINALIZADO when all visas complete is repeated in multiple API routes — a change to business rules requires updating all locations

### 3. `CalendarioView` is a 712-line monolith
- `components/CalendarioView.tsx` (or similar) is oversized and handles too many responsibilities — should be split into subcomponents

---

## Scaling Limits

### 1. Readable ID generation via RPC / COUNT
- Human-readable IDs (`GJ-XXXX`, `VISA-XXXX`, etc.) are generated by counting existing rows, which can produce duplicates under concurrent inserts — needs a sequence or DB-level generation

---

## Dependencies at Risk

### 1. Supabase package version coupling
- `@supabase/ssr` and `@supabase/supabase-js` versions are tightly coupled — a major version bump to either can break `createServerClient` / `createServiceRoleClient` initialization patterns

---

## Missing Features / Gaps

1. **No pagination** — any list with unbounded growth will eventually be slow
2. **No React error boundaries** — unhandled rendering errors crash the full page
3. **No date validation** — date inputs (appointment dates, seminar dates) lack client/server-side validation for past dates or invalid ranges
