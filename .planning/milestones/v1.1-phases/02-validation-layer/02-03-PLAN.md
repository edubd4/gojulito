  <action>
For each of the three files, make a small but critical change to the success check in the fetch response handler.

**Current pattern in all three files:**
```typescript
if (!res.ok || !json.success) {
  setServerError(json.error ?? 'Error al ...')
  return
}
```

**New pattern (per D-11):**
```typescript
if (!res.ok || json.error) {
  setServerError(json.error ?? 'Error al ...')
  return
}
```

This works because:
- `res.ok` catches HTTP errors (401, 404, 500)
- `json.error` catches validation errors where status is 400 but error message exists
- The server no longer returns `json.success` — it returns `{ data, error }`

**File-specific details:**

**NuevoClienteModal.tsx:**
- Line 175: change `if (!res.ok || !json.success)` to `if (!res.ok || json.error)`
- Line 169: The 409 DUPLICATE_CLIENT check (`json.error === 'DUPLICATE_CLIENT'`) MUST remain BEFORE the general error check. The server still returns `{ error: 'DUPLICATE_CLIENT', message: '...', cliente_existente: {...} }` on 409. This block is already correct — it checks `res.status === 409 && json.error === 'DUPLICATE_CLIENT'` before reaching the general check.
- After the success path, the component likely does `onSuccess(json.cliente)` — change to `onSuccess(json.data)` since the server now returns `{ data: cliente, error: null }` not `{ success: true, cliente: ... }`.

**EditarClienteModal.tsx:**
- Line 195: change `if (!res.ok || !json.success)` to `if (!res.ok || json.error)`
- After success, change `json.cliente` references to `json.data` (the server now returns `{ data: clienteActualizado, error: null }`).

**RegistrarPagoModal.tsx:**
- Line 124: change `if (!res.ok || !json.success)` to `if (!res.ok || json.error)`
- After success, change `json.pago` references to `json.data` (the server now returns `{ data: pago, error: null }`).
